# 平台功能分析与长期运营规划报告

## 1. 核心目标
明确 **QQ机器人** (MusicalBot) 与 **网页平台** (MusicalWeb) 的功能边界。
**核心策略**：**Bot做减法** (专注触达)，**Web做加法** (深度交互)。解决过去“指令复杂劝退用户”和“Repo功能无人问津”的痛点。

## 2. 数据与反馈现状分析

### 📊 用户行为数据 (`StatsDataManager.json`)
*   **高频 (Vital)**: `被动通知` (12w+), `快捷搜剧` (6k+), `日期查询` (3k+)。
*   **低频 (Issues)**: `盲盒Repo` (<100), `订阅管理` (600+)。

### 💡 关键洞察
1.  **订阅机制过重**: 复杂的指令参数（如 `-T -1/-2/-3`）导致用户望而却步，极大限制了活跃用户的转化。
2.  **Repo功能错位**: 在聊天窗口回复大量文字和图片（Repo）不仅交互体验差，且无法有效沉淀为社区内容，导致使用率极低。
3.  **查询习惯**: 虽然Web端已实现查询功能，但Bot端的 `快捷搜索` 仍有保留价值，用于满足“群聊场景下顺手查一下”的即时需求。

## 3. 功能规划与平台分工

### 🤖 QQ机器人: "极简触达终端" (Notification Hub)
**定位**: 不再试图做成全能App，而是作为Web平台的**"触角"**。

| 功能模块 | 现状 | 规划动作 | 预期价值 |
| :--- | :--- | :--- | :--- |
| **消息推送** | 核心功能 | **保留并增强**。消息中嵌入直达Web的深链接 (Deep Link)。 | 维持用户留存的根本，增加Web引流入口。 |
| **订阅指令** | 极其复杂 | **大幅简化/移除**。仅保留 `/订阅 <剧名>` 引导去Web设置，或者完全砍掉管理指令，只负责发消息。 | 降低门槛，消除挫败感。 |
| **快捷查询** | 习惯性使用 | **保留基础版**。仅返回最核心信息（有/无，最低价），详情引导去Web看。 | 满足即时场景，避免Bot输出过长刷屏。 |
| **Repo/评价** | 极其难用 | **完全移除**。引导用户去Web填写。 | 聚焦核心体验，不再做无用功。 |

### 🌐 网页平台: "深度交互与获客中心" (The Platform)
**定位**: 承接所有需要**可视化**、**复杂输入**和**浏览**的需求。

| 功能模块 | 规划详情 | 获客/运营价值 |
| :--- | :--- | :--- |
| **学生票Repo中心** | **(新增)** 交互式作为图/列表。允许用户上传视角图、打分、写长评。 | **内容沉淀**。高质量的避雷/种草内容是吸引新用户（Search Traffic）的最佳手段。 |
| **可视订阅管理** | **(重构)** 登录后，勾选剧目/演员即可订阅。无需记忆任何指令。 | **提升转化**。让小白用户也能轻松用起来，增加对平台的依赖粘性。 |
| **深度查询/筛选** | 卡司同台、历史掉落热力图、价格趋势。 | **专业工具**。留住硬核粉丝 (Power Users)。 |

## 4. 推荐的 UI/UX 用户路径 (User Journey)

### 🚀 路径一：从被动推送到主动贡献 (Retention Loop)
1.  **Bot**: 推送《剧目A》掉落通知。
2.  **User**: 点击链接跳转 **Web** 查看详情。
3.  **Web**: 页面下方展示该场次的 **"历史Repo"** (避雷/推荐)。
4.  **Web**: 用户购票观看后，在Web上提交自己的Repo（上传视角图）。
5.  **Bonus**: 提交Repo获得积分，解锁更多订阅额度。

### 🎣 路径二：极简订阅 (Acquisition Flow)
1.  **User**: 访问 **Web** 首页，看到《剧目B》的海报。
2.  **Web**: 按钮 [🔔 订阅此剧掉落]。
3.  **Web**: 弹窗提示 "绑定QQ机器人接收实时通知" -> 一键复制绑定指令。
4.  **Bot**: 用户发送绑定指令，完成闭环。

## 5. 结论
过去的问题在于**强行在Bot端实现了Web的逻辑**。
未来的方向是**让Bot回归"通知器"的本质，让Web承担"应用"的职责**。

*   **立刻停止**：维护Bot端的Repo指令、复杂的管理指令。
*   **优先开发**：Web端的Repo上传/展示页面、可视化的订阅管理面板。
